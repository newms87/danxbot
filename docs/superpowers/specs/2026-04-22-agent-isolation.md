# Agent Isolation — Strict MCP + Dispatch Profiles + Workspace Dir

**Trello:** https://trello.com/c/7ha2CSpc (Epic)
**Shipped:** 2026-04-22 across 5 phases + a pre-Phase-5 dev-CLI commit.

> **Update (Phase 3 of card `kMQ170Ea`, 2026-04-23):** the "Slack SDK path" referenced below (`src/agent/agent.ts`, in-process `query()`) was retired. The Slack deep-agent now funnels through the same `dispatch()` + `spawnAgent()` path as the poller and `/api/launch`. References to `src/agent/agent.ts` and "in-process SDK query" in the text below describe the state at the time Agent Isolation shipped; the current system has a single CLI-spawn path for every dispatch.

## Problem

Before this epic, danxbot-dispatched claude agents (poller, HTTP `/api/launch`, Slack) shared configuration with the developer's interactive `claude` session:

- Both discovered the same repo-root `.mcp.json`
- Both read `<repo>/.claude/settings.local.json`
- Both saw the same `<repo>/CLAUDE.md` and `<repo>/.claude/rules/*.md`
- Poller ticks mutated `<repo>/.claude/rules/`, `<repo>/.claude/skills/`, `<repo>/.claude/tools/` — writing `danx-*` files into the developer's tracked / working tree

Cross-contamination was fragile. A real failure mode: a trello MCP key renamed in the dev's `.mcp.json` took out danxbot's trello integration because both processes read the same file. Another failure mode: every Slack message spawned an `npx` subprocess for the trello MCP server even though the Slack allowlist denied every trello tool — because the danxbot repo's own `.mcp.json` was auto-loaded by the SDK.

The repo root must belong to the developer for use case #1 (interactive CLI). Danxbot must bring its own rules, MCP servers, env vars, and tool allowlist at dispatch time, with zero influence from the repo root.

## Solution

A strict isolation boundary enforced by THREE mechanisms working together:

### 1. Per-repo workspace directory

Every connected repo gets a generated `<repo>/.danxbot/workspace/` directory. Inside it:

- `.mcp.json` — empty stub `{"mcpServers": {}}`
- `CLAUDE.md` — workspace marker
- `.claude/settings.json` — `{"env": {"DANXBOT_WORKER_PORT": "<port>"}}`
- `.claude/rules/` + `.claude/skills/` + `.claude/tools/` + `.claude/agents/` — populated by the poller's inject pipeline

The workspace is generated by `src/workspace/generate.ts`:
- At worker boot (between `syncSettingsFileOnBoot` and `assertJsonlDirectoryAccess`)
- On every poll tick (top of `_poll`, so config changes propagate without a restart)

Idempotent — unchanged inputs yield zero writes.

### 2. Dispatched agents cwd into the workspace

`src/agent/launcher.ts` spawns claude with `cwd = workspacePath(repoName)` for both docker-headless and host-interactive paths. `src/agent/agent.ts` (Slack SDK path) uses `cwd = workspacePath(repoContext)`. `src/worker/dispatch.ts` resolves resume-session directories from the workspace.

Combined with `--strict-mcp-config` (always applied in `src/agent/claude-invocation.ts`), this ensures the per-dispatch `.mcp.json` is the ONLY MCP config claude reads — user scope, project scope, and the repo-root `.mcp.json` are all ignored.

### 3. Dispatch profiles — named, registry-backed tool surfaces

`src/dispatch/profiles.ts` defines three profiles:

| Profile | Allowlist |
|---------|-----------|
| `poller` | `Read / Glob / Grep / Edit / Write / Bash / TodoWrite / Agent / Task / mcp__trello__*` |
| `http-launch` | `[]` — empty baseline; every request supplies its own surface via `body.allow_tools` |
| `slack` | `Read / Glob / Grep / Bash` — read-only for in-process SDK query |

Every consumer routes through ONE entry point:

```ts
const allowTools = dispatchAllowTools(profileName, bodyOverrides);
```

`resolveProfile` + `mergeProfileWithBody` remain internal implementation details — production callers never import them directly.

## Non-goals (deliberately deferred)

- Per-repo poller tool customization — would require extending `DispatchProfile` with a per-repo slot. Not needed today; the poller profile serves all connected repos.
- MCP server opt-in as a separate field — the allowlist is the single source of truth for both "what tools claude exposes" and "which servers the per-dispatch `.mcp.json` contains" (activation happens automatically from `mcp__<server>__*` entries in the allowlist via `src/agent/resolve-dispatch-tools.ts`).

## Invariants (must remain true)

1. **Dispatched agent cwd = `<repo>/.danxbot/workspace/`.** For every entry point (poller, HTTP launch/resume, Slack). Verified by `src/agent/agent.test.ts`, `src/__tests__/integration/dispatch-pipeline.test.ts`, and parity tests in `src/workspace/generate.test.ts`.
2. **Zero writes from danxbot into `<repo>/.claude/`.** Enforced by `src/poller/index.test.ts` — the "workspace-only" assertion rejects any write outside `<repo>/.danxbot/workspace/.claude/`.
3. **`--strict-mcp-config` always present on dispatched claude invocations.** Enforced by `src/agent/claude-invocation.ts` via `buildClaudeInvocation`. Flag lives in the always-on list alongside `--mcp-config` and `--allowed-tools`.
4. **`--allowed-tools` always present and matches the dispatch profile.** No callsite builds its own allowlist; every one routes through `dispatchAllowTools`.
5. **Workspace port read from `<repo>/.danxbot/.env`** (local dev) or `process.env.DANXBOT_WORKER_PORT` (production compose). The legacy `<repo>/.claude/settings.local.json` read path was removed in Phase 5.
6. **Renaming / removing the trello MCP server from the repo-root `.mcp.json` has zero effect on dispatched agents.** The workspace's per-dispatch MCP config is the only one claude reads.

## Phase 5 changes in detail

1. `src/poller/index.ts` — `buildInjectTargets` (plural → singular `buildInjectTarget`), all per-target loops collapsed to single-call operations on one target (the workspace). `_poll`'s trello-config dual-write reduced to workspace-only. `writeRuleToAll` / `copyRuleToAll` / `copyExecutableToolToAll` abstractions removed — each helper now operates on one target directly.
2. `src/agent/agent.ts` — Slack cwd switched to `workspacePath(repoContext)`; `SLACK_ALLOW_TOOLS` replaced with `dispatchAllowTools("slack")`.
3. `src/dispatch/profiles.ts` — `"slack"` added to `DispatchProfileName`; `SLACK_ALLOW_TOOLS` (formerly in `src/slack/constants.ts`, now deleted) lives in `DISPATCH_PROFILES.slack.allowTools`.
4. `src/repo-context.ts#readWorkerPort` — reads `DANXBOT_WORKER_PORT` from the already-parsed `.danxbot/.env` map; no longer touches `<repo>/.claude/settings.local.json`.
5. `Makefile` — `launch-worker` and `launch-worker-host` extract `DANXBOT_WORKER_PORT` via grep on `<repo>/.danxbot/.env` instead of `jq` on `.claude/settings.local.json`.
6. 3 connected repos' `.danxbot/.env` updated to include `DANXBOT_WORKER_PORT=<5560|5561|5562>`.
7. Danx-* files deleted from `<repo>/.claude/rules/`, `<repo>/.claude/skills/`, `<repo>/.claude/tools/` across platform + gpt-manager. Backup at `~/web/danxbot-bkp/phase5-pre-delete/`. `gpt-manager/.claude/rules/danx-ui.md` (a pre-existing, unrelated rule about the `danx-ui` npm package) preserved.
8. Tests updated: `src/poller/index.test.ts`, `src/repo-context.test.ts`, `src/agent/agent.test.ts`, `src/dispatch/profiles.test.ts`.
9. Docs: `.claude/rules/docker-runtime.md`, `.claude/rules/agent-dispatch.md`, `CLAUDE.md`.

Committing that cleanup is a separate concern in that repo; since Phase 5 stops danxbot from writing to `<repo>/.claude/`, no new files will accumulate.
